# Claude Code adapter 误把 90% 的 `type:"user"` 打成 human · 现场报告

- 日期：2026-05-23
- 仓库：`memory.talk`（branch `main`，HEAD `054b477`）
- 复现命令：
  ```bash
  memory-talk read sess_0a294181-ffc4-4942-8374-8bae539b9e95
  # 前 3 条 round.role 都是 "human"，但内容是 <local-command-caveat> /
  # <command-name>/model</command-name> / <local-command-stdout>
  ```
- 数据目录：`~/.memory-talk/sessions/claude-code/0a/sess_0a294181-…/`
- 上游源：`~/.claude/projects/-Users-zzz-Projects-alisre-abm-worker/0a294181-….jsonl`
- 结论：**adapter 只看顶层 `type` 字段判定 role，把 Claude Code transcript 里 4 种来源的 `type:"user"` 行全部贴上 `role="human" / speaker="user"`。本会话 250 条 `type=user` 中，222 条（89%）是误判，其中 220 条其实是工具回灌（`tool_result`）。**

---

## 1. 现象

`memory-talk read sess_0a294181-ffc4-4942-8374-8bae539b9e95` 读出 525 个 round，最前 3 条都是 `role: "human"`：

```jsonl
{"idx":1,…,"speaker":"user","role":"human","text":"<local-command-caveat>Caveat: The messages below were generated by the user while running local commands. DO NOT respond to these messages or otherwise consider them in your response unless the user explicitly asks you to.</local-command-caveat>", …}
{"idx":2,…,"speaker":"user","role":"human","text":"<command-name>/model</command-name>\n            <command-message>model</command-message>\n            <command-args>claude-opus-4-7</command-args>", …}
{"idx":3,…,"speaker":"user","role":"human","text":"<local-command-stdout>Set model to Opus 4.7 for this session</local-command-stdout>", …}
```

这 3 条都不是人输入。它们是 Claude Code CLI 在用户敲 `/model claude-opus-4-7` 这一刻自动注入的「caveat + command 调用 + command stdout」三件套——所有 slash 命令都会在 transcript 里产生这种结构。

用户的真·首条输入在 `idx=4`，是一段 JSON 错误日志加上「这个报错 gateway/apps/openclaw/api/v2/instances 分析一下原因」。

---

## 2. 误判规模

把本 session 上游 JSONL（`0a294181-….jsonl`）里所有 `type:"user"` 行按稳定信号分类：

```
total type=user rows: 250

  220  tool_result          ← 模型自己发起的工具调用，回灌成 user-side message
   26  real-human           ← 真·人输入
    1  isMeta(caveat)       ← <local-command-caveat>，且顶层 isMeta=true
    1  slash-command-invoke ← <command-name>/model</command-name>...
    1  slash-command-stdout ← <local-command-stdout>...</local-command-stdout>
    1  other                ← [{"type":"text","text":"[Request interrupted by user]"}]
```

当前 adapter 一律打 `role="human"`，所以：

| 真实来源 | 数量 | 当前标签 | 是否误判 |
|---|---:|---|:---:|
| tool_result | 220 | human | ❌ |
| real-human | 26 | human | ✓ |
| isMeta caveat | 1 | human | ❌ |
| slash-command-invoke | 1 | human | △（是用户敲的命令，但不是消息） |
| slash-command-stdout | 1 | human | ❌ |
| `[Request interrupted by user]` | 1 | human | △（状态消息） |

**误判率 = 222 / 250 ≈ 89%**，其中 220 条 `tool_result` 数量上是大头，1 条 `isMeta` 是用户最先注意到的「前 3 条不对劲」中的第 1 条。

---

## 3. 根因

`memorytalk/adapters/claude_code.py:183-205` 的 `_parse_message`：

```python
def _parse_message(self, msg: dict) -> RoundInput | None:
    msg_type = msg.get("type")
    if msg_type not in ("user", "assistant"):
        return None
    role = "human" if msg_type == "user" else "assistant"
    speaker = "user" if msg_type == "user" else "assistant"
    ...
```

判定粒度太粗。Claude Code transcript 里 `type` 是 **Anthropic Messages API 的 message role**（user / assistant），它表达的是「这条 message 进 LLM 时挂在哪一侧」，**不是「说话人」**。在 user 侧 payload 里，CLI 会把以下东西统统打包成 `type:"user"`：

1. **真人键盘输入** —— `message.content` 是字符串或 text block，无其它特征字段。
2. **tool_result** —— 模型上一步发起的工具调用回灌；`message.content[0].type == "tool_result"`，且顶层有 `toolUseResult` 字段（含 `stdout` / `stderr` / `interrupted` 等）。
3. **slash 命令三件套** —— CLI 在用户敲 `/xxx` 那一刻注入 caveat / `<command-name>` / `<local-command-stdout>`；caveat 同时带 `isMeta: true`。
4. **状态消息** —— 如 `[Request interrupted by user]`。

adapter 没有看任何子信号（`isMeta` / `message.content[0].type` / 顶层 `toolUseResult` / content 文本前缀），全部归 `human`。

`role` 字段在 schema 里是 `str | None` 自由文本（`memorytalk/schemas/session.py:37`），且下游 `cli/_format.py` 在 `_round_row` 上方的注释里**显式预期**会出现 `TOOL` / `SYSTEM` 值：

```python
# Role is uppercased (``HUMAN`` / ``ASSISTANT`` / ``TOOL`` / ``SYSTEM``)
# so it reads like a log-level tag and visually separates from content.
```

也就是说渲染层早就为 4 类 role 设计好了，**只是 adapter 从来没真的产出过 `tool` / `system`**。

---

## 4. 下游影响

`role` 字段的 4 个下游消费点（grep 自代码）：

| 文件:行 | 用法 | 误判后果 |
|---|---|---|
| `memorytalk/provider/lancedb.py:51` | `rounds` 表 schema 字段 | LanceDB 列里 90% 的 role=human 是假的 |
| `memorytalk/service/search.py:242,310,337` | hit / context_round 元数据 | search 结果里大量 HUMAN 实际是 tool 输出 |
| `memorytalk/cli/_format.py:120,159,384` | 渲染 `[N HUMAN]` 标签 | 视觉上读不出工具回灌 vs 真人提问 |
| `memorytalk/service/cards.py:196` | 喂给 card 提炼的 round 上下文 | LLM 生成 card 时归因错乱，把工具输出当人话提炼 |

特别危险的是 **search + cards 这条链**：
- search 召回带 `role:"human"` 的 tool_result 段落
- card 命令把这段交给 LLM，LLM 以为是用户原话
- 提炼出的 card.summary 把工具 stdout 当成「用户的提问 / 决策」存进去
- 未来 recall 反复召回这条假 card，污染不可逆

---

## 5. 每类来源的稳定判定信号

判定都不需要正则猜测，源 JSONL 顶层或 content 头部就有明确字段：

| 来源 | 信号（按优先级从上往下） | 应得 role / speaker |
|---|---|---|
| assistant | 顶层 `type == "assistant"` | `assistant` / `assistant` |
| tool_result | 顶层 `type == "user"` 且（有 `toolUseResult` 字段 **或** `message.content[0].type == "tool_result"`） | `tool` / `tool` |
| caveat / 系统注入 | 顶层 `isMeta is True` | `system` / `harness` |
| slash 命令注入 | `message.content` 是 str，lstrip 后以 `<command-name>` / `<local-command-stdout>` / `<local-command-caveat>` 开头 | `system` / `harness` |
| 真人输入 | 以上都不命中的 `type == "user"` | `human` / `user` |

`toolUseResult` 这个顶层字段是最稳的 tool_result 信号——它由 CLI 写入，跟 content 结构解耦，即使将来 Claude Code 改 content 形状也不会丢。

## 6. 修复方案

### 6.1 改 adapter（治本，必做）

`memorytalk/adapters/claude_code.py:_parse_message` 里在 `msg_type=="user"` 分支按 §5 的优先级分类。代码改动局限在一个文件，约 15 行。`role: str | None` 是自由文本，新增 `tool` / `system` 完全向后兼容；新会话同步即正确。

### 6.2 既存 session 的回填（重要）

`rounds.jsonl` 是 append-only 已固化输出，下面两种方法**都不能**修旧数据：

- `memory-talk sync`：cursor 已经走过，跳过已存在 round，根本不会回头改。
- `memory-talk rebuild`：只重建 SQLite + LanceDB 派生索引，不重写 rounds.jsonl 里的 role 字段。

正确做法是加一个 **reclassify 子命令**：

```
memory-talk reclassify --source claude-code [--session sess_xxx | --all]
```

遍历 `~/.memory-talk/sessions/claude-code/**/rounds.jsonl`，按新分类规则**原地**重写 `role` / `speaker` 字段（保 `round_id` / `idx` / `timestamp` / `content` / `content_hash` 完全不变，cards 通过 round_id 关联，所以不破坏外链），完成后跑一次 `rebuild` 让派生索引对齐。

> 不能用"删了重 sync"绕过：cards 通过 round_id 引用具体 round，删 session 会让 cards 失去来源链接。

### 6.3 加固化测试

把上面 5 类样本（caveat / command-name / command-stdout / real-human / tool_result / `[Request interrupted by user]`）做成 fixture 放进 `memorytalk/tests/adapters/claude_code/`，断言 `_parse_message` 输出正确的 (role, speaker)。防止以后 Claude Code transcript 格式升级时悄悄退化。

---

## 7. 可选治标补丁（不替代 §6）

如果想立刻不让用户在 search 结果里看到「HUMAN 标签 + `<local-command-stdout>` 内容」的尴尬场景，可以在 `cli/_format.py:_round_row` 里做一个临时兜底：role == "human" 但 text 以 `<local-command-` 或 `<command-name>` 开头时，渲染为 `SYSTEM`。

**仅修视觉，不能修排序、召回、card 归因**。建议只在 §6.1 上线后到 §6.2 跑完之间的过渡期临时启用。

## 9. 一句话总结

`type` 是 Anthropic Messages API 的 role（user/assistant 二元），不是 transcript 的「说话人」；在 user-side payload 上叠了 tool_result / harness 注入 / 真人输入三种东西，adapter 必须看子信号才能正确归类——目前 89% 的 `human` 标签都是错的。
