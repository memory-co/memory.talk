# Works (v5) — memory harness

> **v5 重新定位 memory.talk:定位上,记忆管理该有自己的 harness(memory harness),而不是 executor 里贴的一块。**
> 今天所有 agent harness 都是 **executor harness**(为「把任务做完」而生);记忆在它们里是贴上去的副产物。
>
> **但这版不手搓 harness。** v5 交付的是一个 **memory system**——记忆管理的完整能力(结晶 / 治理 / 召回 + 数据 + 接口),设计成**可被不同 harness 嵌入**:**当前嵌 CC(Claude Code)当宿主**;未来实现专职 **memory harness** 时,同一套 system 原样嵌进去。memory harness 是**北极星定位,本版不实现**(见 §7)。

> 状态:**设计中**。本篇是 v5 的立意文(positioning),只定方向、不定实现。机制 / 数据 / 接口细节由后续 v5 文档展开。v5 是一次**彻底重做**,会大幅改写 v4 的表层;但 v4 的**底料**(被治理的问题图 + 以写代读)保留(见 §8)。

---

## 0. 一句话

> **executor harness 的 loop 为「任务做完了吗」而转;memory harness 的 loop 为「这套记忆还真不真、精不精、新不新、召得回吗」而转。**
> v5 立后者这个定位,但落地为一个**可嵌入的 memory system**(先嵌 CC 用起来);专职 memory harness 留待未来(§7)。

## 1. 来龙去脉:LLM 是全局知识的结晶,个人知识还没人结晶

**一个 LLM,本质上是「全人类公共知识」的结构化结晶。** 预训练就是一次巨大的、离线的**结晶**:把人类写下来的海量文本压进权重,成为一种**结构化、可查询**的形态。所以 LLM 看上去「什么都懂」——它是**集体 / 全局**那一层知识的晶体。

但知识有**两层**:

| | 全局知识 | 个人知识 |
|---|---|---|
| 是什么 | 人类公开写下来的、共有的 | 只属于你的:这个项目的决定、这套代码的脾气、你踩过的坑、你的偏好与上下文 |
| 在哪 | 已**结晶**进 LLM 权重 | 散在原始日志 / 笔记 / 会话里,**没结晶** |
| 谁来结晶 | 预训练(一次性、离线、集体) | **没人**——这正是缺口 |

**LLM 解决了全局那层,个人这层没人解决。** 个人知识既**不在权重里**(模型从没见过你的私有上下文),也**没有自己的「训练步」**——没有哪个过程,对你的经验做了「预训练对全人类文本」做的那件事:**原始经验 → 蒸馏 → 结构化 → 结晶**。

**executor harness 没解决它,只是绕过去。** 它对个人知识的办法是**塞原始上下文**:每次会话重述项目背景、堆一个 `CLAUDE.md`、对原始日志跑 RAG。但这是**取原料,不是结晶**——

- 喂进 context / RAG 的是**未蒸馏的生料**,不是晶体;
- 每次会话**重读生料**,什么都没沉淀成结构;
- 个人知识于是**永远停在原料态**:重复、矛盾、只长不剪(呼应 §4 的「烂掉」)。

就像把生文本反复塞进 prompt、却从不拿它**训练**——**个人知识从没拿到它的「训练步」。**

**这就是 memory harness 的根本定位:给个人知识补上那一步结晶。** 它对**你的经验**做「预训练对全人类文本」做的事——持续、在线、私有地,把原始经验结晶成一张**被治理的问题图**(v4 的 card / position / IBIS)。

> **LLM = 全局知识的晶体**(冻结、离线、集体);**memory harness = 个人知识的结晶器**(在线、持续、私有)。两层互补:executor 用着全局晶体(LLM),却**缺**个人晶体——今天个人这层还是生料,memory harness 把它补齐。

## 2. 边界:不越模型上限;只结晶,不取代

两条硬边界,免得 v5 跑偏成另一个东西:

**① 上限 = 模型本身的上限。** memory harness 的一切智能——提炼、治理、调和矛盾、召回——都**借模型来做**,所以它的质量**封顶在模型的能力**上。**当前方案不做任何「超越模型上限」的优化。** 它让模型**记得更好**(喂对材料 + 喂对结构),不让模型**想得超出它本来的限**:赢的是「带着好个人记忆的模型」对「带着生料 / 没记忆的模型」——两者都被同一个模型上限罩着。模型涨,这套记忆水涨船高;但记忆**不替模型补脑**。

**② 个人结晶 ≠ LLM 结晶,且不求取代。** §1 那个类比说的是**角色**(原料 → 结构化晶体),**不是机制**。两者机制完全不同:

| | LLM 结晶(全局) | memory harness 结晶(个人) |
|---|---|---|
| 机制 | 梯度下降、压进权重(神经) | 蒸馏进**被治理的问题图**(符号 / 结构化 / 文件) |
| 形态 | 不可读的连续权重 | 可读、可寻址、可治理的卡 / 边 |
| 时机 | 一次性、离线 | 持续、在线 |

所以**不追求各种「取代性」特性**:不 fine-tune、不训权重、不把记忆做成「你的小模型」,也不想让个人晶体去**替代 / 吞并 / 跟模型竞争**。两层晶体是**互补**(全局 + 个人),不是替身——memory harness 只管把个人这层结晶好,交给模型用。

## 3. 两种 harness:executor vs memory

**executor harness** —— Claude Code、Cursor、Codex、Devin…… 今天的 agent harness 全是这一类。它的 loop 为**完成一个任务**而存在:
```
感知任务 → 规划 → 动作(读文件 / 改代码 / 跑命令)→ 观察 → 重复,直到目标达成
```
记忆在这套 loop 里**是配角**:context window、`CLAUDE.md`、一个 RAG 索引、一个「memory 工具」——存在的意义是**让当前这个任务做得更好**。任务一闭环,它就不再关心记忆。

**memory harness** —— v5 要立的这一类。它的 loop 不为任何用户任务而转,而是为**维护一套知识**而转:
```
摄入经验 → 提炼 → 治理(去重 / 合并 / 调和矛盾 / 衰减 / 修剪)→ 巩固 → 召回,永不闭环
```
它**不执行用户的任务**;它读的是**经验**(executor 跑完的 session),写的是**信念**(被治理的问题图);它对那套知识有**权属**和**职责**。

| | executor harness | memory harness |
|---|---|---|
| loop 为何而转 | 把这个任务做完 | 让这套记忆真 / 精 / 新 / 可召回 |
| 时间尺度 | 一次任务 / 一次会话(episodic) | 跨会话、持续(corpus 级) |
| 读什么 / 写什么 | 读文件、写代码、跑命令 | 读经验、写信念(distill / merge / decay / link) |
| 成功指标 | 任务闭环 | 记忆质量(后续定义,见 §9) |
| 记忆是 | 配角(读的缓存) | 主角(要养的花园) |

## 4. 为什么 executor harness 管不好记忆

不是工程没做到位,是**结构性**的:

- **没有谁的「本职」是记忆。** executor 的激励是「赶紧做完」。反思、提炼、去重、调和矛盾、决定该忘什么——这些在任务压力下都是**会被跳过的开销**。系统里没有一个角色把记忆当 KPI。
- **激励冲突。** 任务驱动的 loop 永远会对记忆卫生**欠投资**;只有一个**成功指标本身就是记忆质量**的 loop,才会去做这些脏活。
- **时间尺度不对。** 巩固、衰减、调和矛盾发生在任务**之间 / 之上**,作用在**整个 corpus** 上,而不是某次任务里。episodic 的 executor 天然碰不到这个尺度。
- **把记忆当只读缓存。** executor 把记忆当**读**的(retrieval),从不当**养**的(curation)。于是记忆**烂掉**:重复、漂移、自相矛盾、只长不剪。

结论:记忆管理不该是 executor 里的一个**工具 / 一步**,而该是一个**独立 harness** 的全部。

## 5. memory harness 是什么(北极星;本版不实现,落地见 §7)

一个 harness,**唯一职责是管理记忆**。它**不是**「也会记事的 agent」——它的主 loop **本身就是**记忆管理。

**它的活(provisional loop):**
- **摄入(Ingest)** —— 把 executor 跑完的 session / 经验拉进来。
- **提炼(Distill)** —— **以写代读**:逐 round 读经验,就地标出问题 / 答案(v4 的 card / mark)。
- **治理(Govern)** —— 去重、合并、连边、调和矛盾、给过时信念踩票 / 衰减、修剪。让图**又真又精**。
- **巩固 / 反思(Consolidate)** —— 周期性重组:聚类、归纳、提出更高阶的问题。
- **召回(Recall)** —— 按需把相关记忆喂回 executor(无意识召回 + 有意识检索)。

executor 的 loop 会闭环;**memory harness 的 loop 永不闭环**——corpus 永远在被养。

**它不做什么(边界):**
- **不是 executor 的 context 加载器**——召回是它**对外提供的服务**,不是它的目的。
- **不是被动的 store / RAG**——它有自己的 loop,会**主动**对 corpus 动手。
- **不做用户任务**——它从不闭环任何用户目标;它若「动作」,动的也只是**记忆本身**。

---

## 6. 它跟 executor harness 的关系

**对等的两层,不是主从。** executor 做事、memory 管知识;memory 是**持久层**,executor 是**易逝的客户端**。

```
   executor harness(们)            memory harness
   ┌──────────────┐               ┌────────────────────┐
   │ 做任务、跑会话 │  ── 交经验 ──▶ │ 摄入 → 提炼 → 治理   │
   │              │  ◀─ 要召回 ── │ → 巩固 →(对外)召回 │
   └──────────────┘               └────────────────────┘
   多个、跨工具/机器/人              一个、共享、持久
```

- **多对一**:多个 executor(跨 Claude Code / Codex / 不同机器 / 不同人)喂同一个 memory harness、也从它召回。
- **协议是边界**:executor 怎么**交接经验**、怎么**请求召回**,是一份明确契约(后续文档)。executor 不直接改记忆,只**提交经验** + **请求召回**;改记忆是 memory harness 的权属。

## 7. 落地策略:memory system 先行,harness 后置

**不直接手搓 memory harness。** §5 的那个自主 loop 是**北极星**;v5 实际造的是它的**内核**——一个 **memory system**:

- **memory system = 记忆管理的完整能力,harness 无关**:结晶(摄入 / 提炼)、治理(去重 / 合并 / 调和 / 衰减)、巩固、召回,加上数据(问题图)与接口(CLI / API / hooks)。**「做什么」全在 system 里;「何时做、谁驱动」留给宿主 harness。**
- **可被不同 harness 嵌入**:同一套 system,谁嵌它,谁就获得记忆管理能力。嵌入面是一份**明确契约**(system 暴露哪些动作 / 事件 / 查询,宿主怎么驱动它们——后续文档)。
- **当前宿主 = CC(Claude Code)**:经 CLI / hooks / skills 嵌进 CC——CC 的会话产生经验、CC 驱动结晶与治理动作、CC 的 hook 做召回。**用现成的 executor harness 当宿主,先把 system 用起来、磨出契约。**
- **未来宿主 = 专职 memory harness**:等它实现时(**本版不做**),同一套 system 原样嵌入——harness 补的只是 §5 那个自主 loop(何时摄入 / 何时巩固 / 预算 / 权属),**system 不用重写**。

| | memory system(v5 造) | 宿主 harness(嵌它的) |
|---|---|---|
| 管什么 | 记忆的**能力与数据**:结晶 / 治理 / 召回、问题图、接口 | **驱动**:何时做、做多少、谁授权 |
| 当前 | 本版交付 | **CC**(现成,经 CLI / hooks / skills) |
| 未来 | 原样复用 | **memory harness**(§5 的自主 loop,本版不实现) |

> 这样分层的赌注:**记忆管理的难点在能力(system),不在驱动(harness)。** 先在 CC 里把能力磨真,harness 到时候只是换一个更专职的驱动器;反过来先手搓 harness、能力却是空的,就是空转的 loop。

## 8. 跟 v4 的关系:承接底料,重做 system

- **v4 造的是底料**:被治理的**问题图**(card = 问题、position = 答案、credence 现算、IBIS 边)+ **写路径**(以写代读的 mark)+ 读 / 搜 / 召回 + file-canonical 存储。**这些留着**——它们是 memory harness 要管理的**材料**。
- **v5 造的是 memory system**:把上面那套底料重组成**一套完整、可嵌入的记忆管理能力**(结晶 / 治理 / 召回 + 嵌入契约),先嵌 CC 运转;以及「认真把这个定位当真」之后引出的**重新架构**(所以 v5「完全不同」,会大改 v4 表层)。§5 的专职 harness 是它未来的宿主之一,**本版不实现**。

一句话:**v4 = 那套记忆;v5 = 把「记忆管理」做成完整 system(先嵌 CC);memory harness = 北极星宿主(未来)。**

## 9. 待定(后续 v5 文档展开)

- **嵌入契约**:memory system 暴露哪些动作 / 事件 / 查询;宿主怎么驱动。CC 宿主怎么嵌(CLI / hooks / skills 的分工)。
- **loop 与触发**:事件驱动(session 关闭即提炼)?定时巩固?预算上限?三者怎么配——**在 CC 宿主里先用 hooks / 手动命令近似,专职 harness 的自主 loop 后置**。
- **自主与权属**:哪些 system 动作可以无人监督地做(合并 / 衰减 / 修剪),哪些要 human / 宿主在环。
- **executor ↔ memory 协议**:交接经验 + 请求召回的契约;边界面。
- **自主下的治理**:v4 的纪律是「别让 AI 自评惊讶、交给检索」;v5 要把同样的纪律**用在 curator 自己身上**——别让一个自主的管家**把 corpus 越管越烂**。
- **指标**:「好记忆」量什么(真实性 / 精简度 / 召回命中率 / 矛盾数 / 陈旧度)。
- **多 executor / 多源**:不同来源的经验怎么调和。
